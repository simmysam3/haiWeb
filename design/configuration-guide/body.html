Using This Guide
§0How To Use This Guide

This guide serves three journeys. Read the path that matches yours. Throughout, Tier refers to the conformance tier defined in §1, and Planned marks specified-but-unshipped behavior.

  • Journey A — Build from scratch. You are writing your own agent in your own stack and need only to conform to the HAIWAVE protocol. You consume @haiwave/client-sdk for the contracts (or reimplement them) and ignore the reference scaffold. Read: §1 → §2 → §3 → §6.7 → §10 → §11.
  • Journey B — Baseline and extend. You fork the reference agent and override specific Tier 2 pieces — a different LLM, a Snowflake transaction store, custom recommendations. Read: all sections, in order.
  • Journey C — Deploy as-is. You run the reference Docker image and change only configuration, environment, and branding. Read: §3 → §4 → §5.1 → §5.2 → §6 → §7 → §8 → §9 → §11.

Tier Model

HAIWAVE ships as three layered packages. Each layer builds only on the ones to its left, so you can take a lower layer without the ones above it — the SDK does not pull in the reference agent. Use the layer that fits your work. This section only defines the vocabulary; the capabilities are enumerated in §5 (for B and C) and §10 (for A).

// build order — strictly acyclic (enforced by tsc -b project references)
@haiwave/protocol  →  @haiwave/client-sdk  →  @haiwave/reference-agent  →  root app
   (wire types)        (prescribed surface)      (reference scaffold)         (composition)
1.1The Three Tiers

Tier 0 — Wire And Transport. The protocol message schemas (@haiwave/protocol) plus transport and auth: the outbound client (HaiWaveClient, TokenManager), the MCP (Model Context Protocol) server, JWT (JSON Web Token) validation, and the PROTO-3 callback HMAC (Hash-based Message Authentication Code). You cannot change Tier 0 without leaving the network.

Tier 1 — Prescribed Conformance. The interfaces and invariants every agent must honor: the chat pipeline phases, the intent taxonomy, the role floor, the write-confirmation gate, the transaction-touching chains, the data-disclosure contracts. Extend at the declared extension points; never remove or bypass them.

Tier 2 — Reference Scaffold. Everything in @haiwave/reference-agent: the reference data layer (a single-file DuckDB by default), the Claude and Gemini adapters, the local embedding model, the reference pricing, lead-time, and recommendation logic, and the UI. Each piece is freely replaceable behind the Tier 1 interfaces.

Data store is your choice. DuckDB is only the quick-start default — single-file and in-process, so the reference agent runs with no external database. Production deployments point the DataStore interface at the data platform they already run (Snowflake or any other) and reuse that investment; the agent is not tied to DuckDB. The heavy native dependencies (DuckDB, embeddings, the LLM vendor SDKs, document parsers) live only in @haiwave/reference-agent — the root app and @haiwave/client-sdk stay free of them, and guard tests pin that boundary.

SDK Quick-Start

The minimum to get a Tier 0 + Tier 1 conformant agent running from @haiwave/client-sdk.

2.1Prerequisites
  • Node.js 22+.
  • A reasoning key. An Anthropic key (ANTHROPIC_API_KEY) or a Gemini key (GEMINI_API_KEY).
  • Network credentials from §3: PARTICIPANT_ID, AGENT_ID, KEYCLOAK_CLIENT_ID, KEYCLOAK_CLIENT_SECRET.
  • @haiwave/client-sdk. Obtained inside the Free Agent source downloaded from the HAIWAVE console (a workspace package consumed from that source, not a public npm install). It declares @haiwave/protocol as a peer dependency, satisfied by the protocol copy that ships hoisted in the same source package (§11).
2.2Tier 0 — Transport And Auth

The SDK provides the transport primitives. Wire them in this order:

  1. Token management. Instantiate TokenManager with your Keycloak client credentials and KEYCLOAK_TOKEN_ENDPOINT. It handles client-credentials token acquisition, caching, and refresh.
  2. Central client. Instantiate HaiWaveClient (typed by the IHaiWaveClient contract) with the TokenManager and HAICORE_BASE_URL. This is your outbound channel to HAIWAVE Central; it attaches the bearer token to every call.
  3. MCP server. Build your inbound MCP server with createMcpServer; per-request session state comes from createSessionState.
  4. Inbound auth. Gate the /mcp route with validateJwt (configured by McpAuthConfig). Enforcement is controlled by MCP_AUTH_REQUIRED; supply KEYCLOAK_JWKS_URI / KEYCLOAK_ISSUER for RS256 in production.
  5. Origin assertion. Call assertHaiCoreOrigin on inbound requests so only HAIWAVE Central can route to your agent (§6.7).
  6. Register tools and resources on the MCP server (the protocol tool surface is in §10.2).

2 · SDK Quick-Start
Minimal wiring — the two halves: outbound to Central, inbound MCP server.
import { TokenManager, HaiWaveClient, createMcpServer }
  from '@haiwave/client-sdk';

// Outbound: client-credentials token manager + the authenticated Central client.
const tokenManager = new TokenManager({
  tokenEndpoint, clientId, clientSecret,
  scopes: ['gofish:query', 'gofish:respond', 'catalog:query' /* full set in .env.example */],
});
const central = new HaiWaveClient({
  baseUrl, participantId, agentId, forwardOnBehalfOf: false,
}, tokenManager);
// `central` now attaches a fresh bearer token to every haiCore call.

// Inbound: the MCP server haiCore calls into. Register the tools you expose (§10.2).
const { mcpServer, sessionState } = createMcpServer({
  serverInfo, capabilities, registrations,
});
// Mount mcpServer on a Streamable HTTP transport at POST /mcp — see src/mcp/server.ts.
2.3Tier 1 — Chat Baseline

If your agent exposes the conversational surface, wire the prescribed chat baseline from @haiwave/client-sdk:

  1. Pipeline. Run each turn through runMessageTurn, which executes the eight prescribed phases in order (§10.6).
  2. Sessions. Manage conversational state with SessionManager / ChatSession.
  3. Routing. Route against the VALID_INTENTS taxonomy using QueryRouter (with an IQueryFastPath for the regex fast path).
  4. Role floor. Enforce role gating via hasMinimumRole; the quote_accept intent has a hard trader floor.
  5. Chains. Register the 11 prescribed transaction-touching chains in the PromptChainRegistry.
  6. Write gate. Preserve the pending-confirmation state machine (handlePendingConfirmation plus the arming in finalizeTurn) so no write executes without explicit acknowledgment.

2.4 Deploy-as-is shortcut (C). If you are not modifying code, skip §2 and go to §4. The SDK quick-start is for Journeys B and A; chain extension points belong to §5.4 (B) and §10 (A).

Verify withnpx vitest run packages/client-sdk/src/__tests__/conformance/

Network Registration

Joining the HAIWAVE network is a gated, human-reviewed process, not a self-service signup. Every participant is screened and explicitly admitted by a HAIWAVE gatekeeper before any credentials are issued. Credentials — and therefore the ability to download the agent and operate in network mode — arrive only on approval. The flow has three stages.

  1. Public application. A prospective supplier completes the form on the marketing site (/join). This creates an inert holding-pen record (status = pending_approval); no participant, Keycloak user, or session is created yet, and the response is a uniform acknowledgement that leaks no screening outcome. It collects contact, legal entity name and website, headquarters country (an ISO 3166-1 alpha-2 code — the screening key), headquarters address, and at least one business identifier (Tax ID or DUNS, the Data Universal Numbering System ID).
  2. Automated jurisdiction screening. HAIWAVE Central assigns a risk tier from the headquarters country alone — blocked (sanctioned jurisdiction), elevated (outside the gatekeeper's domestic jurisdiction), or standard (domestic). Screening is a signal for the human gatekeeper, not an automatic gate.
  3. Gatekeeper adjudication. A gatekeeper reviews pending applications at Admin › Registration requests (/account/admin/registrations) and approves or rejects (both audited). Rejection requires a reason and tombstones the record (PII partly redacted) for later re-invitation. Approving a blocked-tier application requires an explicit, audited override.

On approval. An idempotent provisioning step creates your participant record (participant_id / agent_id), creates a Keycloak user for console access (a one-time setup requiring password reset plus TOTP, a Time-based One-Time Password, enrollment), and emails a single-use activation link. From there you can sign in to the console, download the agent source or Docker image, retrieve your agent's Keycloak client credentials, and proceed with the network-mode deployment in §4. The six trust tiers assigned to your counterparties are covered in §6.2.

Deploy the Reference Agent
4.1Source And Image

After §3 approval, download the reference agent from the HAIWAVE console download page (Account › Agent Software) — the definitive source for the Free Agent client source package, the @haiwave/client-sdk, and this configuration guide. The reference image is a standard node:22 build. The container ships its config as a seed template and mounts the runtime config on a volume (VOLUME /data/config). On first boot the entrypoint copies the seed into the mounted volume and never overwrites existing files, so an image upgrade cannot clobber your runtime customizations. Set CONFIG_DIR=/data/config to point the agent at the mounted volume.

PlannedFIPS-hardened agent image. A FIPS-hardened agent image is on the roadmap. The FIPS-validated images that exist today are for HAIWAVE Central, not the partner-deployed agent.

4.2Standalone vs. Network Mode
  • Standalone. A local AI assistant for your team. No HAIWAVE credentials, no network participation, no Go Fish, no manifest exchange.
  • Network. Everything standalone does, plus Keycloak authentication, network registration, heartbeats, Go Fish discovery, manifest exchange, and order flow. Requires PARTICIPANT_ID, AGENT_ID, and Keycloak client credentials.

PlannedAutomatic mode selection. The intended design selects the mode from the presence of network credentials. Today the switch is not wired: the chat surfaces assume network mode, and PARTICIPANT_ID / AGENT_ID carry placeholder defaults when unset. Treat the reference agent as network-oriented; coordinate with your HAIWAVE contact for a pure standalone deployment.

5 · Extend & Customize
5.2LLM Provider And Cost

The agent uses a hosted LLM for all reasoning; you pay the provider directly (HAIWAVE neither provides keys nor proxies calls). Select the reasoning provider with LLM_PROVIDER (claude default, or gemini) and the embedding provider with EMBEDDING_PROVIDER (local default — in-process and key-free — or gemini). Default models are claude-sonnet-4-6 and gemini-2.5-flash. Both adapters are supported configurations.

Cost is driven by query volume and catalog size. The agent uses a two-tier query router: a tight regex fast-path classifies a narrow set of unambiguous, high-frequency requests (order lookups, ERP status, confirmations) without a model call; everything else falls through to a lightweight LLM classification step. The fast-path is deliberately narrow, so savings are real but bounded. The session and daily token caps are hard cost ceilings.

Support scope. The reference Claude / Gemini configuration is covered by HAIWAVE support. Modifications beyond provider selection — different models, additional providers, custom prompt structures — fall outside that scope.

5.4Tier 2 Extension Chains

The reference agent registers the 11 prescribed transaction-touching chains (§10.6) plus its Tier-2 local-data chains — inventory_check, pricing_lookup, product_specs, product_compatibility, cross_reference, standards_lookup, document_search, company_policy, erp_status, general, classification_status — into the PromptChainRegistry, all wired by registerAllChains in @haiwave/reference-agent (src/chat/chains/). A Journey-B reader can replace any chain by registering a different PromptChain for the same intent, provided it preserves the chain's prescribed payloads and write-confirmation behavior. The Tier-2 chains are reference-only and freely replaceable.

Security & Access Posture
6.1Managed Access Model Overview

In network mode your agent participates in the HAIWAVE Managed Access Security Model. haiCore handles authentication, threat detection, and intent classification before a request reaches your agent; your agent handles tier-appropriate response construction and field visibility from the trust-tier label carried on each routed request.

6.2Trust Tier Configuration

Each counterparty relationship has a trust tier that bounds maximum disclosure.

TierNameAssignment
0InternalYour own users/agents · automatic
1PremierYou explicitly designate a counterparty
2Trading PairAutomatic when a trading pair is established
3ConnectionAutomatic when a bilateral connection is acknowledged
4QualifiedAutomatic for network-verified participants
5PublicNon-verified · no access

Only Premier requires explicit action. Assign per counterparty via PUT /api/v1/relationships/{owner_id}/counterparties/{counterparty_id}/tier; the same endpoint with { "action": "remove_override" } reverts to the system-assigned tier. Console: Account Management › Trading Partners (/account/partners); cross-modality behavior under Settings › Trust Posture.

6.3Tier Permission Profile

Part of your Company Operations Manifest; controls four dimensions: intent screening (on/off per tier), response type (direct_data / clean_room / configurable_blend), document sharing (direct / public_url_only / withheld), and intent-class permissions per tier. Defaults ship with the platform; override in your manifest or at runtime via GET/PUT /api/v1/participants/{participant_id}/tier-config.

The 11 managed-access intent classes are: Product Discovery, Pricing & Commercial Terms, Inventory & Availability, Order Management, Shipping & Logistics, Returns & Disputes, Compliance & Documentation, Account & Relationship, Technical Support, Vendor Qualification, Strategic Partnership. For each class at each tier the value is Permitted (your agent responds) or Screened (haiCore returns a standard message; your agent never sees the request). Tiers 0–2 permit all 11 by default; higher tiers screen progressively more. Unknown intents are dispositioned per tier via unknown_disposition: screen (reject + log; default for Connection/Qualified) or accept (forward best-effort + log; default for Internal/Premier/Trading Pair). All unknowns are logged. These access-control intent classes are distinct from the chat router's VALID_INTENTS taxonomy (§10.6).

6.5Review Queue

Unknown and flagged requests surface under Account Management › Data Cleansing (/account/data-cleansing). Review them, and submit promising unknowns to HAIWAVE for evaluation as new known intents (also via GET /api/v1/.../review-queue + submit / dismiss). Recognized submissions are added to the known library network-wide.

6.6Audit Log And Anomaly Monitoring

The managed-access audit log records every classified request (timestamp, sender, tier, zone, intent, permitted/screened, unknown disposition) and is available at GET /api/v1/participants/{participant_id}/access-log (threat entries are summary-only). HAIWAVE also runs network-level anomaly monitoring (about 15 categories) with three escalation levels:

  • ALERT about 3× baseline — logged.
  • THROTTLE about 5× baseline or 100+ products/hr — a full one-hour block, auto-resolving.
  • SUSPEND about 10× baseline or repeated threats — blocked pending review.

Thresholds are platform-level, not company-configurable; pull your own events via GET /api/v1/.../anomaly-events.

6.7Inbound Request Validation (Agent-Side)

Your agent must accept inbound requests only from haiCore — this is the structural guarantee that every request has passed authentication, threat detection, and intent classification first. The SDK provides assertHaiCoreOrigin for the origin check. The IP allowlist and service-credential schemas are in the Protocol Specification (mirrored by @haiwave/protocol).

PlannedInbound IP allowlist + service credential. The intended design adds two mechanisms on the inbound path: an IP allowlist (HAIWAVE_ALLOWED_IPS, accept only Central's published ranges) and a service credential (HAIWAVE_SERVICE_CREDENTIAL, validated on every routed request). Neither is read by the current reference agent. Today the only inbound auth is the admin API key (ADMIN_API_KEY via X-Admin-Key) guarding /admin/*, plus optional JWT on /mcp (MCP_AUTH_REQUIRED). Do not assume the network-facing inbound path is authenticated at the agent until this ships; a custom agent (§10) should implement equivalent validation per the Protocol Specification regardless.

6.8Approval Gateway

Optional, disabled by default. The approval gateway pauses a high-stakes action until an explicit decision comes back, instead of letting the agent act on its own. It gates three action types — trading_pair_connection, invoice_approval, and order_release (the last only when the order value exceeds APPROVAL_GATEWAY__ORDER_RELEASE_THRESHOLD). With the gateway off (APPROVAL_GATEWAY__ENABLED=false, the default), these actions proceed without gating.

  • Native. The agent holds the pending approval in-process — pair it with a local approval UI or a simple hold-and-resume. This is the mode when no MCP endpoint is set.
  • MCP. The agent delegates the decision to an external gateway at APPROVAL_GATEWAY__MCP_ENDPOINT. The decision returns asynchronously to POST /approvals/callback, whose body is verified with an HMAC (Hash-based Message Authentication Code) over the raw request bytes using APPROVAL_GATEWAY__CALLBACK_SECRET — the PROTO-3 guard the SDK exposes as verifyCallbackSignature.

While a request is pending the originating action is staged and resumes on the decision. A pending approval not decided within APPROVAL_GATEWAY__EXPIRY_MINUTES (default 1440 — 24 hours) is swept (every __EXPIRY_CHECK_INTERVAL_MINUTES, default 5) and resumed as rejected; the gateway polls decision status every __POLL_INTERVAL_SECONDS (default 180).

The contract is Tier 1: IApprovalGateway (submit/poll), IApprovalRepository, and one IApprovalResumeHandler per approval type — all in @haiwave/client-sdk. The reference agent supplies the concretes (NativeApprovalGateway / McpApprovalGateway, ApprovalRepository, and the trading-pair / invoice / order-release resume handlers); swap any behind those interfaces.

8.1Phantom Demand Data Source

Phantom Demand is a one-time, on-demand, non-binding feasibility spot check: a buyer-side product's BOM (Bill of Materials) is exploded and each in-scope vendor agent is probed. Wire it from local data:

  • BOM. Lives in the boms / bom_lines DuckDB tables; your agent serves one level via the get_bom_for_sku MCP tool while Central drives the multi-tier walk. Seed from your ERP (scripts/seed-phantom-demand-bom.ts).
  • Lead-time history. The p50/p75/p90 figures derive from the local transaction_events table (scripts/seed-lead-time-history.ts) — there is no separate order-history source.
  • Internal capacity. Served via the get_internal_capacity MCP tool through an optional MES bridge: MES_ENABLED (default false), MES_CONFIG ({endpoint_url, auth_scheme, credential_ref, work_center_mapping}), and SKU_PICKER_SCOPE (published_only / full_catalog). These can also be set centrally via PATCH /api/v1/agents/{agent_id}/config (Admin › Agent Config), falling back to the env vars if Central is unreachable.

PlannedReal MES driver. The reference agent ships a mock MES driver returning stubbed capacity so the integration can be exercised end-to-end. A production MES adapter (SAP, Oracle, custom) is a per-customer integration, not included.

When you answer a phantom-demand probe as a seller, on-hand disclosure is gated by the requester's manifest tier: premier → exact, standard → sufficient / insufficient, basic → not_disclosed. Console: Sonar Observe › Phantom Demand (/account/sonar/observations).

8.2Continuous Monitoring (Watchers)

A Watcher is a standing, recurring monitor against a counterparty's supply-chain signals. On each run Central polls each in-scope vendor agent and records lead-time, capacity, and delivery signals, emitting drift events into your compliance change feed. A watcher scope sets counterparties, signal_types, skus (empty [] = company-wide watch), depth_limit (1–8), and optional drift_thresholds; cadence is manual_only / daily / weekly / monthly / event_triggered. The five signal types are lead_time_distribution, published_lead_time, quoted_lead_time, capacity_utilization_band, delivery_event. Console: Sonar Observe › Watcher Management (/account/sonar/watchers).

Responder requirement: if you are watched, answer the get_watcher_signals MCP tool (read-only; Central authorizes before the call reaches you).

PlannedDemo-data signals. published_lead_time / quoted_lead_time are synthetic and capacity_utilization_band falls back to a fixed value when its seed table is absent; the drift-to-change-feed pipeline itself is live.

8.3Compliance Data Capture

Each completed audit run produces a Compliance Snapshot (the traversal plus per-(vendor, product) attribute samples), diffed across runs into typed Change Events. Two attribute kinds are sampled today: lead_time_days (from your local lead-time intelligence) and certification_status (valid / not_held, read from the product's certifications array). As a responder, answer the get_compliance_attributes MCP call (wired automatically; no console setting). Cadence follows your audit-run template (Sonar Audit › Audit Management, /account/sonar/audit); daily for high-change supply chains, weekly for stable ones.

9 · Provenance & Key Management
09
Journeys B · C
Provenance & Key Management

Product provenance (Sonar Audit › Product Provenance, /account/provenance) is a read-only, product-led composition browser assembled from declared origin manifests. It shows declared composition while audits (§8.3) verify it. There is no partner action beyond browsing — the data comes from the origin manifests you and your counterparties declare.

Provenance keys (Sonar Audit › Key Management, /account/provenance-keys) are scoped access credentials that control which sensitive origin and permission fields flow during provenance traversal (not product certifications). A key declares required_fields[] / requested_fields[], plus friendly_name, purpose, expires_at, enabled, and revoked. There are two roles: a generator issues a key (the raw value is revealed once; any re-reveal is audited), and an installer previews and accepts it, committing to share those fields. Narrowing a key's required fields starts a 14-day grace window — installations move to non-compliant rather than dropping.

Independent mechanisms. Provenance keys and the compliance certification_status attribute (§8.3) merely share the Sonar Audit console area — there is no code path tying a key installation to the certification-status response. Agent-side, your only dependency is having products.certifications populated and origin manifests declared so an installed key has data to release; keys and installations are stored centrally.

10.6Chat Conformance Invariants

If your agent offers the conversational surface, it must preserve these six invariants (enforced behaviorally by the conformance kit in §10.7):

  1. Full intent taxonomy. Route against the complete VALID_INTENTS set (22 intents: order, quote, catalog, gofish, inventory, pricing, specs, resolution, audit, and so on) — no narrowing of the taxonomy; unknown intents fall back to general.
  2. Write-confirmation gate. No write executes without explicit user acknowledgment (the pending-confirmation state machine).
  3. Role ACL with a quote_accept → trader floor. No lowering; the hierarchy is operator < trader < admin.
  4. The 11 transaction-touching chains at their prescribed payloads and call sequences: orderConfirmChain, quoteRequestChain, quoteAcceptChain, quoteStatusChain, quoteCompareChain, orderLookupChain, gofishQueryChain, catalogSearchChain, productResolutionChain, sourceAuditChain, manifestStatusChain.
  5. Eight pipeline phases in order: guardTurn → handlePendingConfirmation → detectContextFilter → prepareRouterInput → routeAndRemapIntents → executeSubIntents → saveSearchContext → finalizeTurn — with the scope, haiwave_only, and write-confirmation gates unconditional.
  6. WebSocket message shape { type, content_panel, intent }.

Everything else — which LLM, which local-data backend, UI, auth provider — is unconstrained. The executable companion to these invariants is the conformance kit (§10.7), which pins both Tier 0 and Tier 1 behaviorally.
