1 · Tier Model
01
Foundations
Tier Model

HAIWAVE ships as three layered packages. The dependency arrow points only left; an adopter consumes the surface appropriate to their journey.

// dependency direction — strictly acyclic (enforced by tsc -b project references)
@haiwave/protocol  →  @haiwave/client-sdk  →  @haiwave/reference-agent  →  root app
   (wire types)        (prescribed surface)      (reference scaffold)         (composition)
1.1The three tiers

Tier 0 — Wire & transport. The protocol message schemas (@haiwave/protocol) plus transport and auth (HaiWaveClient, TokenManager, the MCP server, JWT validation, the PROTO-3 callback HMAC). You cannot change Tier 0 without leaving the network.

Tier 1 — Prescribed conformance. The interfaces and invariants every agent must honor: the chat pipeline phases, the intent taxonomy, the role floor, the write-confirmation gate, the transaction-touching chains, the data-disclosure contracts. Extend at the declared extension points; never remove or bypass them.

Tier 2 — Reference scaffold. Everything in @haiwave/reference-agent: the DuckDB data layer, the Claude/Gemini adapters, the local embedding model, the reference pricing/lead-time/recommendation logic, the UI. Freely replaceable behind the Tier 1 interfaces.

Heavy-dep firewall. The root app and @haiwave/client-sdk are free of the six heavy native deps (DuckDB, embeddings, the LLM vendor SDKs, doc parsers) — those live only in @haiwave/reference-agent. Guard tests pin this.

2 · SDK Quick-Start
02
Journeys A · B
SDK Quick-Start
Prerequisites — what to have before wiring.
  • Node.js 22+.
  • A reasoning key. ANTHROPIC_API_KEY or GEMINI_API_KEY.
  • Network credentials from §3: PARTICIPANT_ID, AGENT_ID, KEYCLOAK_CLIENT_ID, KEYCLOAK_CLIENT_SECRET.
  • @haiwave/client-sdk — obtained inside the Free Agent source from the console download page (a workspace package, not a public registry install). It declares @haiwave/protocol as a peer dependency satisfied by the in-source copy.
2.1Tier 0 — minimal wiring

Two halves: the outbound client to HAIWAVE Central, and the inbound MCP server haiCore calls into.

import { TokenManager, HaiWaveClient, createMcpServer }
  from '@haiwave/client-sdk';

// Outbound: client-credentials token + Central client.
const tokenManager = new TokenManager({
  tokenEndpoint, clientId, clientSecret, scopes });
const central = new HaiWaveClient({
  baseUrl, participantId, agentId, forwardOnBehalfOf: false
}, tokenManager);

// Inbound: the MCP server; register the tools you expose (§10.2).
const { mcpServer, sessionState } = createMcpServer({
  serverInfo, capabilities, registrations });
// Mount on Streamable HTTP at POST /mcp — see src/mcp/server.ts.
2.2Tier 1 — chat baseline
  • Pipeline. Run each turn through runMessageTurn (the eight prescribed phases, §10.6).
  • Routing. Route against VALID_INTENTS with QueryRouter; register the 11 prescribed chains in PromptChainRegistry.
  • Gates. Enforce the quote_accept → trader role floor and the pending-confirmation write gate.
Verify withnpx vitest run packages/client-sdk/src/__tests__/conformance/

3 · Network Registration
03
Journeys A · B · C
Network Registration

Joining the HAIWAVE network is a gated, human-reviewed process — not self-service. Every participant is screened and explicitly admitted before credentials are issued.

  1. Public application. A prospective supplier completes the form at /join. This creates an inert holding-pen record (status = pending_approval); no participant, Keycloak user, or session is created yet, and the response is a uniform acknowledgement. Collects contact, legal entity + website, HQ country (ISO 3166-1 alpha-2, the screening key), HQ address, and at least one business identifier (Tax ID or DUNS).
  2. Automated jurisdiction screening. Central assigns a risk tier from the HQ country alone — blocked / elevated / standard. Screening is a signal for the human gatekeeper, not an automatic gate.
  3. Gatekeeper adjudication. A gatekeeper reviews pending applications at Admin › Registration requests and approves or rejects (both audited). Approving a blocked-tier application requires an explicit, audited override.

On approval. An idempotent provisioning step creates your participant record (participant_id / agent_id), creates a console user (one-time password reset + TOTP), and emails a single-use activation link. From there you can sign in, download the agent, and retrieve your agent's Keycloak client credentials.

4 · Deploy the Reference Agent
04
Journeys B · C
Deploy the Reference Agent

After approval, download the Free Agent source or the pre-built Docker image from the console (Account › Agent Software — the definitive source for the SDK, the agent, and this guide). The container ships config as a seed template and mounts the runtime config on a volume (VOLUME /data/config); set CONFIG_DIR=/data/config.

4.1Standalone vs. network mode

Standalone — a local AI assistant; no credentials, no network participation. Network — adds Keycloak auth, registration, heartbeats, Go Fish, manifest exchange, and order flow; requires the identity + Keycloak credentials. Planned automatic mode selection from credential presence is not yet wired — treat the agent as network-oriented.

4.2Deployment steps
  1. Keys. Set ANTHROPIC_API_KEY (or GEMINI_API_KEY) and, for network mode, the identity + Keycloak credentials + HAICORE_BASE_URL.
  2. Data. Set DUCKDB_PATH (omit for in-memory) and load your catalog.
  3. Config. Set manifest posture, pricing defaults, and business documents; restart.
  4. Register. The agent authenticates, bootstraps network config, and becomes discoverable; the product scanner classifies the catalog.

6 · Security & Access Posture
06
Journeys A · B · C
Security & Access Posture

In network mode your agent participates in the Managed Access Security Model: haiCore handles authentication, threat detection, and intent classification before a request reaches you; your agent handles tier-appropriate response construction and field visibility from the trust-tier label on each routed request.

6.1Trust tiers
TierNameAssignment
0InternalAutomatic — your own users/agents
1PremierYou explicitly designate
2Trading PairAutomatic on trading pair
3ConnectionAutomatic on bilateral connection
4QualifiedAutomatic for verified participants
5PublicNon-verified · no access

Only Premier requires explicit action (Account Management › Trading Partners). A per-tier permission profile in your manifest controls intent screening, response type (direct_data/clean_room/configurable_blend), document sharing, and which of the 11 managed-access intent classes are permitted.

PlannedInbound IP allowlist + service credential. HAIWAVE_ALLOWED_IPS and HAIWAVE_SERVICE_CREDENTIAL are specified but not yet read by the reference agent. A custom agent (§10) must implement equivalent origin + service-credential validation; the SDK provides assertHaiCoreOrigin and verifyCallbackSignature today.

7 · Manifest Configuration
07
Journeys B · C
Manifest Configuration
7.1Pricing disclosure

Configured under pricing_disclosure; layered on top of Managed Access (which decides whether a pricing intent reaches you; this decides how much detail appears). Levels: full · range · rank_only · hidden. A network trust floor applies first; the more restrictive of (floor, your config) wins — you can tighten but never loosen. The price_basis label is Tier 1 (verified by the confirmation handler); only the arithmetic is swappable. Account Management › Manifests.

7.2N-best recommendations

Opt-in under narrowing_and_recommendations (enable_recommendations, max_recommendations_per_response default 3, types upgrade/alternative/accessory/bundle). Swappable via IRecommendationEngine; the buyer agent caps displayed recommendations at 5 total per query.

7.3Brand preferences & posture

Buyer-side under brand_preferences (preferred_brands, weight_boost 0–0.30, allow_override). Manifest-exchange posture knobs are Tier 2; the envelope shape and posture vocabulary are Tier 1 (POSTURE_CONFIG_PATH). Planned behavioral brand-preference suggestions are not yet implemented.

8 · Monitoring Configuration
08
Journeys B · C
Monitoring Configuration
8.1Phantom demand

A one-time, on-demand feasibility spot check: a buyer-side BOM is exploded and each in-scope vendor agent probed. Serve BOM via get_bom_for_sku (from boms/bom_lines); lead-time history derives from transaction_events; internal capacity via get_internal_capacity through an optional MES bridge (MES_ENABLED, MES_CONFIG, SKU_PICKER_SCOPE). Planned the reference agent ships a mock MES driver; a production adapter is a per-customer integration. Sonar Observe › Phantom Demand.

8.2Continuous monitoring (Watchers)

A Watcher is a standing monitor against a counterparty's supply-chain signals; Central polls each in-scope vendor agent and records lead-time/capacity/delivery signals, emitting drift into your compliance feed. Responder requirement: if watched, answer get_watcher_signals (read-only; Central authorizes first). Sonar Observe › Watcher Management.

8.3Compliance data capture

Each audit run produces a Compliance Snapshot diffed into typed Change Events; two attribute kinds are sampled today (lead_time_days, certification_status). As a responder, answer get_compliance_attributes (wired automatically). Sonar Audit › Audit Management.

9 · Provenance & Key Management
09
Journeys B · C
Provenance & Key Management

Product provenance (Sonar Audit › Product Provenance) is a read-only composition browser assembled from declared origin manifests; audits verify what it shows. No partner action beyond browsing.

Provenance keys (Sonar Audit › Key Management) are scoped credentials that control which sensitive origin/permission fields flow during traversal. A key declares required_fields[] / requested_fields[] (+ friendly name, purpose, expiry, enabled/revoked). A generator issues a key (raw value revealed once; re-reveal audited); an installer previews and accepts it. Narrowing a key's required fields opens a 14-day grace window.

Independent mechanisms. Provenance keys and the compliance certification_status attribute merely share the Sonar Audit console area — no code path ties a key installation to a certification response. Your only agent-side dependency is having products.certifications populated and origin manifests declared.

10 · Build From Scratch (SDK Path)
10.3Chat conformance invariants
  1. Full intent taxonomy. Route against the complete VALID_INTENTS (22) — no narrowing; unknown intents fall back to general.
  2. Write-confirmation gate. No write executes without explicit acknowledgment.
  3. Role floor. quote_accept → trader; hierarchy operator < trader < admin.
  4. The 11 transaction-touching chains at prescribed payloads/sequence.
  5. Eight pipeline phases in order, gates unconditional.
  6. WebSocket frame { type, content_panel, intent }.
10.4Conformance kit

The executable companion: the kit shipped in @haiwave/client-sdk pins these Tier-0 + Tier-1 invariants behaviorally. Self-verify before requesting participation.

Runnpx vitest run packages/client-sdk/src/__tests__/conformance/

Inbound validation. Implement origin + service-credential validation per the protocol spec regardless of the reference agent's current Planned gap. The SDK provides assertHaiCoreOrigin and verifyCallbackSignature (PROTO-3 HMAC).
