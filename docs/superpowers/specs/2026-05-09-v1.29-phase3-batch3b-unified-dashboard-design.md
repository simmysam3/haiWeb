## 4. Page composition

`page.tsx` is a Server Component. A single `loadDashboard()` helper does parallel `fetchJson` over the four BFF endpoints (cross-modality, activity, throttled count, latest-run summaries per modality), mirroring `audit/dashboard/page.tsx`'s pattern (`cookies().toString()` + host detection + `cache: 'no-store'`).

The page renders four sections top-to-bottom.

### 4.1 Section 1 — Header strip (`_components/header-strip.tsx`)

Server component. 4-tile grid + 2 right-aligned controls.

- **Tile 1**: Total partners observed (count from cross-modality response)
- **Tile 2**: Last run time across modalities (max of audit/type2/phantom-demand `triggered_at`)
- **Tile 3**: Throttled runs — reuses existing `<ThrottledRunsPanel>` from `src/components/sonar/throttled-runs-panel.tsx`
- **Tile 4**: Failed runs (last 30d) — counted at BFF from the same merged runs list activity feed uses, filtered to `status='failed' AND triggered_at >= now-30d`
- **Refresh button** (client) — calls `router.refresh()` to re-render the server component
- **Run all CTA** (client) — see §6.1

### 4.2 Section 2 — Modality lens (`_components/modality-lens.tsx`)

3-up grid: `grid grid-cols-1 lg:grid-cols-3 gap-4`. Three server components:

- `AuditPostureCard`: compliant / partial / non-compliant counts + top 3 gap classes + "View details →" link to `/account/sonar/audit/dashboard`
- `PhantomDemandCard`: latest probe response rate + capacity confidence histogram (sparkline) + "View details →" link to `/account/sonar/phantom-demand/dashboard`
- `Type2SignalsCard`: lead time medians + capacity bands by tier-1 partner + "View details →" link to `/account/sonar/type2/dashboard`

Each card is a thin shell over the BFF payload — no extra fetches.

### 4.3 Section 3 — Cross-modality risk view (`_components/cross-modality-table.tsx`)

`'use client'`. Receives the cross-modality payload as a prop from the server component (no second fetch).

- Columns: Partner · Audit · Phantom Demand · Type 2 capacity · Type 2 lead time p90 · Risk
- Risk pill renders color + visible text label (`green/normal`, `yellow/elevated`, `red/critical`) per spec §7.5 WCAG 2.1 AA
- Client `useState` for sort column + direction; default sort by `risk_score` desc
- Empty cells render as `—` when the modality has no data for that partner
- Empty table state: "No partners observed yet."

### 4.4 Section 4 — Activity feed (`_components/activity-feed.tsx`)

`'use client'`. SWR with `refreshInterval: 30000`.

- SWR key: `/api/account/sonar/dashboard/activity` so Run-all CTA can call `mutate()` on it
- Renders ~30 rows: `[modality icon] [status pill] [partner name?] [triggered_at relative]` with deep link to the modality run-detail page
- Empty state: "No recent runs across modalities."
- Error state: SWR `error` renders inline "Failed to load activity" with retry hint

## 5. Cross-modality data join + risk-score derivation

### 5.1 Partner set

The "tier-1 partners" axis is the union of:

- `audit_run_results.counterparty_id` from the latest completed audit run
- Type 2 LatestSnapshot's per-counterparty rows from the latest complete Type 2 run
- Phantom demand window's tier-1 counterparties from the latest window's aggregate

**Display name resolution**: each modality already returns `counterparty_name` / `vendor_name`; BFF picks the first non-null. If all null, fall back to `partner_id`.

### 5.2 Per-modality weight derivation

Each weight ∈ [0, 1]:

```
audit_non_compliant_weight  = non_compliant_sku_count / total_sku_count   // for that partner in latest audit run
                              || 0.25  if no audit data for this partner

phantom_demand_low_response_weight = 1 - response_rate
                              || 0.25  if no phantom demand data for this partner

type2_capacity_pressure_weight =
  match latest type2 capacity_band:
    'low'      → 0.0
    'moderate' → 0.33
    'high'     → 0.67
    'at_capacity' → 1.0
    null/missing → 0.25
```

Note: `total_sku_count === 0` for a partner that exists in audit but has zero SKUs returns `0` (not `0.25`) — that's a present-and-empty signal, not missing data.

### 5.3 Aggregate

```
risk_score = audit_w * 0.4 + pd_w * 0.3 + t2_w * 0.3      // ∈ [0, 1]

color = risk_score < 0.33 ? 'green'
      : risk_score < 0.67 ? 'yellow'
      : 'red'

label = match color:
  'green' → 'normal'
  'yellow' → 'elevated'
  'red' → 'critical'
```

### 5.4 Missing-data floor rationale

A partner with no data in modality X gets weight `0.25` for X (not `0`). This:

- Surfaces *some* signal so completely unobserved partners don't blend into the green floor
- Avoids false-positiving fresh networks: a partner with no data in any modality scores `0.25 * 1.0 = 0.25` → still green
- Lets a single severe modality push to yellow on its own: e.g. at-capacity type 2 (1.0) with two missing modalities scores `0.475` → yellow
- Two missing + 50% non-compliant audit: `0.5*0.4 + 0.25*0.3 + 0.25*0.3 = 0.35` → yellow

### 5.5 Validation scenarios (test targets)

| Scenario | Expected color |
|---|---|
| Partner present in all 3 modalities, all healthy (0/0/0) | green |
| Partner red in all 3 modalities (1/1/1) | red |
| Partner with no data in any modality (0.25/0.25/0.25) | green (0.25) |
| Partner with at-capacity type 2 only, missing PD + audit (0.25/0.25/1.0) | yellow (0.475) |
| Partner with 50% non-compliant audit + at-capacity type 2, missing PD (0.5/0.25/1.0) | yellow (0.575) |
| Partner with 90% non-compliant audit only, missing PD + type 2 (0.9/0.25/0.25) | yellow (0.51) |
| Threshold boundary: `risk_score === 0.33` | yellow (`<` is strict) |
| Threshold boundary: `risk_score === 0.67` | red (`<` is strict) |

### 5.6 BFF response shape — `/api/account/sonar/dashboard/cross-modality`

```typescript
{
  partners: Array<{
    partner_id: string;
    partner_name: string;
    audit: {
      compliant: number;
      partial: number;
      non_compliant: number;
      total: number;
    } | null;
    phantom_demand: {
      response_rate: number;
      window_id: string;
    } | null;
    type2: {
      capacity_band: 'low' | 'moderate' | 'high' | 'at_capacity';
      lead_time_p90_days: number | null;
    } | null;
    risk_score: number;       // 0..1
    risk_color: 'green' | 'yellow' | 'red';
    risk_label: 'normal' | 'elevated' | 'critical';
  }>;
  generated_at: string;       // ISO timestamp
}
```

The server-side computes `risk_score`, `risk_color`, `risk_label` once so the table component does no math. Client-side sort by risk = sort numeric `risk_score`.

## 6. Run all CTA

### 6.1 Behavior

Endpoint: `POST /api/account/sonar/templates/run-all`.

1. Auth-check the participant via `withHaiCore`
2. List enabled templates: `client.listRunTemplates({ enabled: true })` (Batch 3a method)
3. `Promise.allSettled(templates.map(t => client.triggerRunTemplate(t.template_id)))`
4. Return `{ triggered: Array<{ template_id, run_id }>, failed: Array<{ template_id, error_message }>, total: number }`
5. HTTP status: `200` always (even partial failure) — body's `failed[]` is the failure surface

### 6.2 Client component (`_components/run-all-button.tsx`)

`'use client'`.

- Disabled while `isPending` (prevents double-click from same tab)
- On success: toast `"Triggered N runs (M failed)"`, calls `mutate('/api/account/sonar/dashboard/activity')` for instant feed refresh
- Empty-templates case: button shows tooltip `"No enabled templates yet — create one in Run Templates"`, button still rendered but disabled

### 6.3 Concurrency caveat (carried forward, not fixed)

Two simultaneous "Run all" clicks from different tabs will produce duplicates — manual triggers bypass the 60s event-debounce in `ScheduledRunService`. Acceptable behavior: a user clicking "Run all" twice means they want runs. `last_run_at` reflects the most recent of the duplicate triggers. Documented as known behavior; no in-flight lock added.

## 7. Template detail run history (Batch 3a carryover)

Endpoint: `GET /api/account/sonar/templates/[id]/runs`.

1. Read template via `client.getRunTemplate(id)` to determine `observation_class`
2. If `audit`: `client.listAuditRuns({ template_id: id, limit: 25 })`
3. If `type2`: `client.listType2Runs({ template_id: id, limit: 25 })`
4. Return `{ runs: Array<{ run_id, status, triggered_at, completed_at?, run_origin }> }`

### 7.1 Open dependency

`listAuditRuns` and `listType2Runs` may not currently accept a `template_id` filter parameter on the BFF or HaiwaveClient methods. From Batch 1's shipped doc, haiCore's `mapRunRow`/`mapRun` includes `run_origin` + `template_id` on the response shape, but Batch 1 didn't document filter param support.

**Plan task should verify and add the filter** at HaiwaveClient + (if missing on the haiCore side) at the route layer. If haiCore-side filtering needs adding, this introduces a haiCore change to the batch — a deliberate exception to the "HaiWeb-only" rule for this one filter param. Decision deferred to plan time after the verification step.

### 7.2 UI integration

`/account/sonar/templates/[id]/page.tsx` (existing 3a page) gets a new `<TemplateRunHistory templateId={id} />` section between the edit form and the `ManualTriggerButton`. Renders a small table: `[run_id] [status pill] [run_origin badge] [triggered_at] [link to run detail]`. Empty state: "This template hasn't been triggered yet."

## 8. Activity feed normalization

`GET /api/account/sonar/dashboard/activity` shape:

```typescript
{
  events: Array<{
    run_id: string;
    modality: 'audit' | 'phantom_demand' | 'type2';
    status: string;            // modality-specific status verbatim
    partner_id: string | null;
    partner_name: string | null;
    triggered_at: string;      // ISO
    completed_at: string | null;
    run_origin: 'ad_hoc' | 'template_manual' | 'template_scheduled' | 'template_event_triggered';
    detail_href: string;       // pre-computed deep link to the modality run-detail page
  }>;
  generated_at: string;
}
```

BFF assembly:

1. Parallel fetch: `client.listAuditRuns({ limit: 20 })`, `client.listType2Runs({ limit: 20 })`, phantom-demand recent windows (TBD which existing endpoint — check at plan time)
2. Normalize each to `ActivityEvent` shape
3. Sort by `triggered_at` desc
4. Slice to top 30
5. Compute `detail_href` per modality:
   - `audit` → `/account/sonar/audit/runs/${run_id}`
   - `type2` → `/account/sonar/type2/runs/${run_id}` (if such a page exists; verify at plan time)
   - `phantom_demand` → `/account/sonar/phantom-demand/reports/${window_id}`

If the limit-30 cap means a flurry of audit runs monopolizes the feed, users drill through the modality-specific runs pages for older events. No pagination on the dashboard feed itself.

## 9. Error handling and empty states

### 9.1 Error handling

Each `loadDashboard()` parallel fetch wrapped in try/catch returning `null` (matches audit dashboard pattern). Section components render degraded states:

- Header strip: tile shows "—" for the missing metric, others render normally
- Modality lens card: shows "Latest data unavailable" stub with retry hint
- Cross-modality table: shows "Cross-modality data unavailable" banner instead of table
- Activity feed: SWR's `error` state shows inline "Failed to load activity" with retry

"Run all" 5xx: toast `"Run all failed — check templates page"`, no auto-retry.

### 9.2 Empty states (first-time user, no runs ever)

- Header strip tiles: 0 / "—" / "Never"
- Modality lens cards: each shows "No runs yet — visit `/audit` / `/phantom-demand` / `/type2` to start"
- Cross-modality table: "No partners observed yet"
- Activity feed: "No recent runs"
- Run all button: disabled with tooltip "No enabled templates yet"

## 12. Plan-time open items — RESOLVED

Investigation done at plan-drafting time:

1. **`template_id` filter on `listAuditRuns` / `listType2Runs`**: Neither HaiwaveClient method nor the haiCore route accepts a `template_id` query parameter. `listAuditRuns` accepts `{ status?, limit? }` (haiCore route at `apps/core/src/routes/audit-runs.ts:148`); `listType2Runs` takes no args (haiCore route at `apps/core/src/routes/type2.ts:77`). **Resolution**: do BFF-side filtering — fetch with `limit=200`, then `runs.filter(r => r.template_id === templateId)` in the BFF route. Captures recent runs for templates that fire frequently; older runs for low-frequency templates won't appear. v1.30 follow-up to add `template_id` to haiCore-side filter.

2. **Phantom-demand recent-windows endpoint**: NO list endpoint exists. `sonar-phantom-demand-reports.ts` exposes only `/latest` (returns single `{ window_id }`), `/:window_id/aggregate`, and `/:window_id/counterparty/:counterparty_id`. **Resolution**: activity feed includes only the *latest* phantom-demand window as a single event (using `/latest` resolution + the window's `generated_at` as the event timestamp). No multi-window history.

3. **Per-partner phantom-demand response rate**: AVAILABLE. The `/sonar/phantom-demand/reports/:window_id/aggregate` response includes `per_counterparty_summary[]` where each entry has `counterparty_participant_id`, `counterparty_display_name`, and `response_rate` (per `packages/protocol/src/phantom-demand/report.ts:27–36`). Cross-modality table can populate the PD column from this array.

4. **`/account/sonar/type2/runs/${run_id}` page**: DOES NOT EXIST. Type 2 has no per-run detail page; per-counterparty drill-down is a modal in the dashboard (`type2/dashboard/_components/per-counterparty-detail.tsx`). **Resolution**: activity feed `detail_href` for type 2 events points to `/account/sonar/type2/dashboard` (the modal opens via the LatestSnapshot grid). Audit events still link to `/account/sonar/audit/runs/${run_id}` (that page exists).

Additional findings during investigation:

- **`listRunTemplates` does NOT accept an `enabled` filter** (it returns all templates). Run-all BFF route must filter `templates.filter(t => t.enabled)` after the call.
- **HaiwaveClient lacks phantom-demand report methods**. Cross-modality BFF route uses `client.fetchRaw('/sonar/phantom-demand/reports/latest')` followed by `client.fetchRaw('/sonar/phantom-demand/reports/${windowId}/aggregate')` (matching the existing pattern at `src/app/api/account/sonar/phantom-demand/route.ts`).
- **`buildPartnerCompliance` already aggregates non-compliant counts per vendor** but does not return total component counts. Plan adds a separate helper `buildPerPartnerAuditWeights(latestRun, results)` in the dashboard's `_lib` that returns `{ vendor_id, non_compliant_count, total_component_count }` rows for risk-weight computation. Existing `buildPartnerCompliance` is left untouched.
- **Type 2 `Type2Result.payload` discrimination by `signal_type`**: capacity band lives in `payload` when `signal_type === 'capacity_utilization_band'`, lead time in `payload` when `signal_type === 'lead_time_distribution'`. The existing `latest-snapshot.tsx`'s `foldByCounterparty()` already does this folding — plan's helper mirrors that logic.
